## New: `getMyBillingStatus()` — one-shot status endpoint

Server-side snapshot of the caller's plan, subscription, storage usage,
and song quota. Use this everywhere you need to gate UI on subscription
state — it replaces the old pattern of stitching `current_plan` +
`subscriptions` + `storage_addons` reads on the client.

```ts
import { getMyBillingStatus } from "@/integrations/cog/billing";

const status = await getMyBillingStatus();
// status.plan, status.is_pro, status.subscription, status.storage, status.song_quota
```

Powered by the `me-billing-status` edge function. `useSubscription()` now
reads from it and exposes the full payload as `state.status`.

## 2026-06-22 payments audit results

Full audit of payments + founder code + referrals + payouts + tracking.

### Backend gaps fixed

- **Monthly payout-draft cron now actually scheduled.** `cog-create-payout-drafts-monthly` (pg_cron, `25 7 1 * *` UTC) sweeps every matured/un-batched cash reward into a draft payout. Previously the function existed in repo but was never applied to the DB, so payable rewards piled up forever and admins never saw drafts.
- **`approve_payout()` enum bug fixed** — it was reading `payout_method` as JSON; it's a `payout_method_kind` enum. Approving any payout would have thrown. Now correctly checks `IS NULL`.
- **Admin RPCs landed:** `admin_finance_summary`, `admin_billing_events`, `admin_list_payouts`, `admin_referrer_ledger`, `admin_fraud_flags` (+ create/resolve), `retry_payout`.
- **OTP toll-fraud rails** (`check_and_record_otp_send` + `otp_send_events` table) are now live — geo allowlist, per-phone/per-IP velocity caps, global daily ceiling.
- **Smoke-test helper** `debug_seed_reward_chain(user, amount, kind)` — service-role only; simulates a paid Pro invoice end-to-end so QA can verify founder + referrer reward minting without real Stripe traffic.

### New endpoint: `getMyFounderStats()`

```ts
import { getMyFounderStats } from "@/integrations/cog/founders";

const stats = await getMyFounderStats();
if (!stats.is_founder) return null;
// stats.code, stats.share_link, stats.redemptions_used / redemptions_cap,
// stats.attributed_count, stats.paying_count,
// stats.earnings.{pending,payable,paid,lifetime}_cents,
// stats.next_draft_date, stats.recent_paid_invoices[]
```

Powered by `me-founder-stats` edge function. Mirror of `getMyReferrals()` but for founder codes. Returns `{ is_founder: false }` for non-founders.

### Existing pipeline (verified end-to-end)

1. Buyer pastes founder code → `create-checkout` atomically claims `claim_founder_code_redemption`, routes to $49 referral price, stamps `attribution_founder_id` on the Stripe subscription metadata.
2. Buyer pastes member referral code → routes to full $100 price, stamps `attribution_referrer_user_id`. One attribution per buyer (UNIQUE constraint).
3. `payments-webhook` writes the attribution row on `checkout.session.completed`.
4. Every `invoice.paid` calls `record_invoice_paid` which mints:
   - Founder: $25/mo months 1–3, $10/mo for life of sub.
   - Member referral: $5/mo for life of sub.
5. Rewards held 30 days (`reward_hold_days`), matured daily by `cog-mature-holds-daily` (07:17 UTC).
6. Day 1 of each month, `cog-create-payout-drafts-monthly` (07:25 UTC) groups matured rewards into draft payouts.
7. Admin reviews via `admin-payouts` edge function (list_drafts/approve/mark_paid). `approve_payout` blocks if recipient has no payout method on file.
8. Refunds + chargebacks reverse pending rewards via `record_invoice_refunded` / `record_chargeback`.

### Live mode

`STRIPE_LIVE_API_KEY` and `PAYMENTS_LIVE_WEBHOOK_SECRET` are provisioned automatically by Lovable once Stripe go-live finishes and the Lovable app is installed on the live account. No code change required when that happens — the webhook URL already routes by `?env=` query param.

## 2026-06-24 — Notifications, reconciliation, transparency

Three world-class hardening additions:

### 1. Lifecycle notifications
- New table `notification_queue` (user_id, kind, payload jsonb, sent_at, attempts).
- DB triggers auto-enqueue on `reward_events` INSERT (`reward_minted`) and status promotions to `payable` / `paid`, and on `payouts` status transitions to `approved` / `paid` (`payout_sent`) / `failed`.
- Edge fn `notify-referral-event` drains the queue every 5 minutes (cron `cog-notify-referral-event`) and sends via Resend connector.
- **Resend connector is not yet linked** — until it is, the function logs "would send" and marks rows as sent so the queue does not grow. Link Resend via Connectors → Resend; no code change required after.

### 2. Reconciliation
- New table `reconciliation_reports` + edge fn `reconcile-billing-events` (daily cron `cog-reconcile-billing-events` at 07:35 UTC, sandbox env).
- Lists Stripe paid invoices in the last 48h, diffs vs `billing_events` where `kind='invoice_paid'`, records drift count and writes a `fraud_flags` row when drift > 0.
- For live mode after go-live, schedule a second cron with `?env=live`.

### 3. Per-recipient event timeline
- Both `me-founder-stats` and `me-referrals` now return `event_timeline[]` (last 50 reward events, newest first) and `payout_method_complete` so dashboards can prompt for missing payout setup.
